## Инструкция по развертке

для развертки необходим docker

```bash
git clone https://github.com/TrixOMG/talent-pay-balance.git

cd talent-pay-balance/

docker compose up -d --build

docker compose exec app php artisan migrate

docker compose exec app php artisan db:seed --class=TestBalanceSeeder
```

далее можно переходить и тестировать:

## Принятые решения

### Защита от двойного подтверждения пополнения

- **Проблема**: Админ может случайно попытаться подтвердить одно и то же пополнение дважды (например, открыл в двух вкладках или нажал кнопку дважды).

- **Решение**: Реализована двухуровневая защита в **_app/Services/WalletService.php_**:
    1.  Проверка статуса перед подтверждением

        ```php
        if ($depositRequest->status !== DepositStatus::PENDING) {
            throw new InvalidArgumentException(
                "Запрос на пополнение #{$depositRequest->id} уже имеет статус {$depositRequest->status}"
            );
        }
        ```

        Если админ попытается подтвердить уже обработанный запрос — получит понятное сообщение об ошибке

    2.  Атомарная операция в транзакции БД **(метод confirmDeposit)**

        где происходит:
        - Блокировка кошелька
        - Изменение баланса
        - Создание транзакции
        - Обновление статуса запроса

    3.  Защита на уровне БД
        - метод _**lockForUpdate()**_ блокирует запись
        - поле **version** поле предотвращает конкурентные изменения
          (идёт проверка версии перед обновлением)

            ```php
            // $wallet до этого был найден по id
            $updated = Wallet::where('id', $wallet->id)
                ->where('version', $wallet->version) // проверка версии
                ->update([
                    'balance' => $balanceAfter,
                    'version' => $wallet->version + 1, // изменение версии
                ]);

                // не нашли $updated с нашей первоначальной версией
                // -> значит кто-то уже изменил -> у нас данные некорректные(протухшие)
                // -> отменяем
                if (!$updated) {
                    throw new RuntimeException('Обнаружена конкурентная модификация кошелька');
                }
            ```

### Защита от параллельных списаний

в своей сути та же, что и в пункте выше, при этом ещё сравнивается сумма списания с балансом на момент списания
(нельзя списать больше чем имеется)

### Возврат средств

В текущей реализации возврат всегда привязан к конкретному списанию через `parent_transaction_id`.

Ограничения и принятые решения:

1.  Нельзя вернуть больше, чем списано т.к. сумма возврата всегда равна сумме исходного списания
2.  Нельзя совершить частичный возврат (только полная сумма)

    частичные возвраты сильно усложняют и логику и аудит, в реальном случае и в зависимости
    от бизнес-логики возможна их реализация

3.  Нельзя сделать возврат дважды **(WalletService, метод refund)**

    ```php
    if ($originalTransaction->hasRefund()) {
        throw new \InvalidArgumentException('По данной операции уже был выполнен возврат');
    }
    ```

    с текущей логикой у одного списания может быть только один возврат

4.  Возврат возможен только для транзакций с типом "Списание"

    ```php
    if ($originalTransaction->type !== TransactionType::WITHDRAWAL) {
        throw new InvalidArgumentException('Возврат возможен только для операций списания');
    }
    ```

5.  Данные для аудита

- parent_transaction_id — прямая ссылка на исходное списание
- В поле metadata сохраняется информация: кто вернул, когда, описание исходной операции

    Таким образом, по цепочке всегда можно проследить: списание -> возврат

6. Возврат защищён от параллельных операций так же, как и другие транзакции в предыдущих пунктах
